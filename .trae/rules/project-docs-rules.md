FitPlus 是一款三分化（Push/Pull/Legs）训练记录 App，基于 React + TypeScript + Vite 构建，数据存储在 localStorage，部署在 GitHub Pages。

## 阅读规则

### 规则 1：版本优先级
- 始终优先阅读最新版本的文档
- 历史版本文档仅在做版本对比或回溯问题时参考
- 需求变更以最新版本需求文档为准，历史版本中已废弃的需求不再适用

### 规则 2：文档阅读顺序
开发新功能或修复 BUG 时，按以下顺序阅读：
1. **目录索引** → 了解整体文档结构
2. **最新需求文档**（01-需求文档）→ 理解功能需求
3. **UI/UE 设计规范**（02-设计文档）→ 确认视觉和交互标准
4. **数据文档**（03-技术文档）→ 了解数据模型和存储结构
5. **最新测试用例**（04-测试文档）→ 了解已有测试覆盖

### 规则 3：文档命名规范
所有文档标题统一格式：`FitPlus - [版本号] [文档类型]`
- 版本号：V1.0 / V1.1 / V1.2.0 / V1.2.1
- 文档类型：需求文档 / 测试用例 / 测试报告 / 设计规范 / 数据文档
- 新增文档时必须遵循此命名规范

### 规则 4：完整研发流程（强制执行）
任何新功能或重大改动必须严格按以下阶段执行，每个阶段有明确的输入、输出和确认节点：

```
阶段 1：需求对齐
  输入：用户原始需求 / BUG 反馈
  输出：完整需求文档（含功能描述、交互说明、边界条件）
  确认：产品负责人确认需求文档 ✅

阶段 2：UI/UE 界面确认
  输入：已确认的需求文档
  输出：UI/UE 设计稿或界面方案确认
  确认：UI 和 UE 同学确认界面方案 ✅

阶段 3：开发
  输入：已确认的需求文档 + UI/UE 方案
  输出：可运行的代码实现
  确认：开发自测通过 ✅

阶段 4：测试
  输入：已确认的需求文档 + 测试用例
  输出：测试报告（含通过/失败记录）
  确认：所有 P0/P1 用例通过 ✅

阶段 5：产品/UI&UE 验收
  输入：测试通过的构建版本
  输出：验收结论（通过 / 打回）
  确认：产品 + UI + UE 均确认通过 ✅

阶段 6：发布
  输入：验收通过的代码
  输出：正式环境部署完成
  确认：CI 构建成功 + 线上可访问 ✅

阶段 7：线上测试回归
  输入：已发布的正式环境
  输出：线上回归测试结论
  确认：核心流程验证通过 ✅
```

**关键原则：**
- 每个阶段必须收到确认后才能进入下一阶段
- 阶段 1 和阶段 2 是前置条件，未经确认不得开始开发
- 阶段 5 验收未通过时，打回至阶段 3 重新开发
- 阶段 7 发现问题时，按 BUG 记录规范（规则 6）记录并进入下一轮修复流程
- 紧急修复（如线上 P0 BUG）可简化流程，但至少保留：开发 → 测试 → 发布 → 线上回归

### 规则 5：需求变更流程
1. 在最新版本需求文档中追加新需求章节（标注日期）
2. 同步更新测试用例文档
3. 开发完成后更新测试报告
4. 更新目录索引文档（如有新文档加入）

### 规则 6：BUG 记录规范
- BUG 统一记录在测试用例文档中，格式为 `BUG-XXX`
- 严重程度：严重 / 一般 / 轻微
- 必须包含：问题描述、复现步骤、根因分析、修复方案、验证方式

### 规则 7：代码仓库
- GitHub：https://github.com/CrazyDave0514/ppl-training
- 分支策略：main（正式环境）
- Commit 规范：Conventional Commits（feat/fix/docs/style/refactor）

### 规则 8：目录索引维护规范
- 新增文档时，在对应分类表格中**新增一行**，禁止新建重复的表格或章节
- 文件夹结构树和版本历史表同步更新
- 禁止使用 append 模式追加重复的分类模块
- 更新目录索引时，必须先读取当前内容，在原有结构上修改
